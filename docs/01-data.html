One canonical preparation, every decision in the open

Every chapter in this suite asks a different question of the same 9,841 addresses — which makes the preparation of those addresses the one place where quiet inconsistency could poison everything downstream. A correlation tested on one cleaning of the data and a regression fitted on a subtly different one are not results about the same population. So preparation is treated here as a first-class analysis step: the raw file is loaded once, every cleaning decision is made once — stated and justified where a reader can disagree with it — and the result is exported as the single starting point for the statistical chapters. The Python exploration deliberately begins from the same raw file instead, walking the data by hand; that is where these decisions are earned rather than asserted.

Cleaning decisions

Four decisions shape the analysis sample — each written down precisely so a reader can push back on it.

Fraud label. Kept both as the numeric flag (for modelling) and as a labelled factor flag_f (for tables and plots). The class balance is retained as-is — the imbalance is a property of the phenomenon, and chapters that are sensitive to it (the logistic regression in particular) address it explicitly rather than resample it away.

ERC20 token types. The two categorical token columns are polluted with placeholder values — "", "0", and whitespace variants — which all mean the same thing: no dominant ERC20 token. They are unified into an explicit "None" category rather than dropped, because “no ERC20 activity” is itself informative.

Missing ERC20 aggregates. 829 addresses have NA across the ERC20 aggregate columns. They are kept here; chapters drop them only where a model requires those variables, and say so. This quirk turns out to be more than a quirk — Chapter 5 returns to it with a finding worth the wait.

Negative balances. 978 addresses show a negative total_ether_balance, an artefact of how the aggregate was computed at extraction time rather than a possible state of an account. They too are kept here and handled where they matter — the linear-regression chapter, which excludes them with justification.

Code 
clean_token <- function(x) {
  x <- trimws(as.character(x))
  ifelse(is.na(x) | x %in% c("", "0", "None"), "None", x)
}

trxs <- trxs %>%
  mutate(
    flag_f = factor(flag, levels = c(0, 1),
                    labels = c("Legitimate", "Fraudulent")),
    erc20_most_sent_token_type = clean_token(erc20_most_sent_token_type),
    erc20_most_rec_token_type  = clean_token(erc20_most_rec_token_type)
  )
