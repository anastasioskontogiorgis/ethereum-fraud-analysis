From behavioural signals to flag probabilities — read with a fraud analyst’s skepticism

Chapter 3 established that which token dominates an address’s inflow is associated with its fraud status. This chapter turns from association to conditional prediction: can transaction behaviour — how much an address transacts, how much of that is token activity, and how quickly it sends — predict the likelihood of a fraud flag? And does the token ecosystem change what those behaviours mean? The goal is inferential understanding of the signals, not a deployable classifier; that distinction ends up mattering more here than in any other chapter.

The analysis sample — and what the missing data quietly revealed

library(dplyr)

library(car)

  mutate(

    token_rec  = relevel(factor(token_rec), ref = "None"),

    erc20_log  = log10(total_erc20_tnxs + 1),
    avgmin_log = log10(avg_min_between_sent_tnx + 1)
  )

n_excl  <- sum(is.na(trxs$total_erc20_tnxs))
excl_fr <- sum(trxs$flag[is.na(trxs$total_erc20_tnxs)])
cat(nrow(lr), "addresses in sample;", sum(lr$flag), "flagged",
    sprintf("(%.1f%%; majority-class baseline accuracy %.1f%%)",
            100 * mean(lr$flag), 100 * (1 - mean(lr$flag))))
9012 addresses in sample; 1350 flagged (15.0%; majority-class baseline accuracy 85.0%)

The model needs the ERC20 aggregates, so the 829 addresses lacking them are excluded — and that routine step surfaced something worth pausing on. The fraud share drops from 22.1% to 15.0%, which is a large move for a 8% exclusion, and the reason is stark: all 829 of the excluded addresses are flagged fraudulent (829/829). In other words, in this dataset “ERC20 data missing” perfectly predicts the label. The most plausible cause is mechanical — the fraud-address list appears to have been assembled separately and joined in, with the ERC20 aggregates never computed for part of that batch — which makes missingness an artefact of dataset construction, not a behaviour of fraudsters. The practical consequences: missingness must never be used as a feature on this data, and everything below speaks for token-active addresses only. Low-frequency token types (< 40 addresses) are pooled into a "Low Count" level so group estimates stay stable, and "None" — no dominant ERC20 inflow — is kept as the meaningful reference category.

Reading the jump carefully

An AUC of 0.98 from a logistic regression should raise an eyebrow before it raises applause, and here the skepticism pays off. Part of the improvement is genuine segment structure — Chapter 3’s association result made that expected. But a decisive part of it is simpler: 2 token levels in this sample coincide perfectly with the fraud flag — Blockwell say NOTSAFU (n = 779) and GSENetwork (n = 80) are flagged fraudulent without exception. Both are known spam-airdrop tokens, and the likely mechanism runs through how the dataset was built: the fraud list appears to draw heavily on addresses swept up in those scam ecosystems, so receiving these tokens is partly a consequence of how the label was constructed rather than an independent behavioural signal. Combined with the missingness artefact found earlier, the message is consistent: this dataset rewards models that memorise its construction.

For anyone taking this analysis toward a production model, three practices follow naturally: validate out-of-time rather than in-sample, so construction artefacts stop paying; ablate token identity and check how much discrimination survives on behaviour alone (here, an AUC of 0.71); and treat token context as segment information whose meaning must be re-learned as scam ecosystems change, not as a stable feature. Read that way, Model 2 is best understood as strong evidence that token context carries signal — and a worked example of why a fraud analyst never deploys an in-sample 0.98.

What this chapter established

Behaviour alone ranks addresses usefully — token-heavy, plain-transaction-light, fast-cadence profiles carry materially higher flag odds — and token context adds real but contaminated lift whose honest assessment requires validation this cross-sectional dataset cannot provide. The estimates speak for token-active addresses, against a label of undocumented provenance, in-sample only; the deployment-grade treatment of this problem — held-out protocols, imbalance handling, cost-sensitive thresholds — is the subject of the dedicated machine-learning project that this suite’s conclusions point toward.
